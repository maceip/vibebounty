## Verified Input Problem

`data/sft/train.jsonl` is not pure label-only data, but its reasoning is short
and outcome-leaky. Local audit on 2026-06-23 found:

- train rows: 17,016, with 15 malformed lines skipped by the audit.
- average pre-JSON rationale: about 214 chars.
- examples use phrases like `It resolved as ...`, which is not information
  available to a live triage analyst.
- class mix is heavily `valid_low`; several rejection labels have little or no
  real corpus support.

The pilot trace files show the intended method:

- assistant content starts with `<think>`.
- average pre-JSON trace is about 1,400 chars.
- traces cite report-specific details before the verdict JSON.

## Improvement Hypothesis

This is not a blind retune. The next run changes the input signal that failed:

1. Replace short outcome rationales with teacher-generated thinking traces.
2. Reject data that lacks `<think>...</think>`, has short reasoning, has parse
   failures, or leaks outcome words such as `resolved as`.
3. Use a corrective trace seed mix so the model sees enough
   `valid_impactful`, `corroborated_surge`, `likely_duplicate`, and
   `out_of_scope` examples to counter the old `valid_low` collapse.
4. Keep deterministic app defenses for classes not represented in the real
   train corpus (`self_inflicted`, `accepted_risk`, some `theoretical_no_poc`).
5. Evaluate with vLLM plus parallel HTTP workers, not serial Transformers.

Expected improvement: higher recall for `valid_impactful` and
`corroborated_surge`, cleaner JSON/verdict formatting, and faster eval. The run
is still only accepted if the held-out eval improves over the prior tune and is
useful for the demo.
